

# Design Decisions

This document captures the architectural decisions made throughout development. It is intentionally informal and serves as the source material for the final project write-up.

Each entry records:

- Decision
- Reasoning
- Alternatives Considered
- Tradeoffs

---

# ADR-001 — Single Operational Datastore

## Decision

Use PostgreSQL with pgvector as both the operational datastore and the vector store.

## Reasoning

The application already requires a relational database for conversations, messages, analyses, and anomalies. Storing embeddings in the same datastore minimizes operational complexity while remaining more than sufficient for the project scale (10,000 conversations).

## Alternatives Considered

- ChromaDB
- Pinecone
- Separate vector database

## Tradeoffs

A dedicated vector database may become appropriate at significantly larger scale, but it would add unnecessary complexity for this project.

---

# ADR-002 — Single AI Provider

## Decision

Use Google AI Studio for both language generation and embeddings.

## Reasoning

A single provider means a single SDK, a single API key, and a simpler deployment experience. Google AI Studio was also explicitly recommended in the assignment and provides both Gemini models and embedding models.

## Alternatives Considered

- Anthropic
- Groq
- OpenAI
- Voyage AI
- Local embedding models

## Tradeoffs

Using multiple providers could optimize individual components, but would increase configuration and operational complexity.

---

# ADR-003 — Structured Conversation Object Stored as JSONB

## Decision

Persist AI-generated conversation understanding as a JSONB document in the `ConversationAnalysis` table.

## Reasoning

The Structured Conversation Object is expected to evolve over time. JSONB allows new fields to be introduced without requiring database schema migrations.

## Alternatives Considered

- Fully normalized relational schema
- Separate columns for every AI-generated attribute

## Tradeoffs

JSONB sacrifices some relational querying in exchange for flexibility and rapid iteration.

---

# ADR-004 — Separation of Raw Data and AI-Derived Data

## Decision

Keep imported source data immutable and store all AI-derived information separately.

## Reasoning

The original dataset should remain an authoritative record of the source material. AI-generated interpretations belong in `ConversationAnalysis` and should never overwrite imported data.

## Alternatives Considered

- Mutating the Conversation table with AI-generated fields

## Tradeoffs

Maintaining separate models adds a small amount of indirection but preserves data lineage, simplifies reprocessing, and allows AI models to evolve independently of the source dataset.

---

# ADR-005 — Developer Experience First

## Decision

Optimize for a one-command startup experience (`make start`) and a landing page that acts as the application's control center.

## Reasoning

A reviewer should be able to clone the repository, start the platform, and discover its capabilities without reading extensive documentation. Infrastructure should remain usable even before AI is configured.

## Tradeoffs

This requires a small amount of additional UI work but significantly improves usability and demonstrates operational thinking.

---

# ADR-006 — Canonical AI Artifact with Relational Projections

## Decision

Treat the Structured Conversation Object as the canonical AI artifact while
building relational projections for downstream consumers.

## Reasoning

The JSONB representation preserves the complete output generated by the LLM and
allows the schema to evolve without requiring database migrations.

Operational components such as anomaly detection should consume relational
projections rather than repeatedly parsing JSON.

## Alternatives Considered

- Query JSONB directly
- Fully normalize all AI-generated data

## Tradeoffs

Maintaining projections introduces one additional processing step but
dramatically simplifies SQL queries while preserving a stable canonical AI
representation.

# ADR-007 — Issue-Centric Analytics

## Decision

Treat issues—not conversations—as the fundamental analytical unit of the
platform.

Each conversation produces a Structured Conversation Object containing zero,
one, or many Issue objects. Those issues are projected into the
ConversationIssue table for analytics and reporting.

## Reasoning

Real customer conversations frequently contain multiple unrelated problems.
Representing issues explicitly avoids forcing a single "primary issue" while
making anomaly detection, reporting, and future retrieval significantly
simpler.

## Alternatives Considered

- Primary issue + secondary issues
- Single issue per conversation

## Tradeoffs

The projection step adds a small amount of processing but produces a much
cleaner analytical model and preserves the complete AI interpretation in the
canonical JSON artifact.

# ADR-008 — Canonical AI Artifact

## Decision

Treat the Structured Conversation Object as the single source of truth for all
AI-derived knowledge.

## Reasoning

Conversation Understanding is responsible for interpreting raw customer
conversations exactly once.

Every downstream capability—analytics, anomaly detection, retrieval,
evaluation, reporting, and future AI workflows—should consume the canonical
Structured Conversation Object or relational projections derived from it.

No downstream component should reinterpret raw conversations independently.

## Alternatives Considered

- Allow each subsystem to parse conversations independently.
- Store separate AI outputs for each downstream component.

## Tradeoffs

This introduces an explicit projection layer but guarantees a single,
consistent interpretation of every conversation while allowing downstream
schemas to evolve independently.

# ADR-009 — Typed AI Contracts

## Decision

Define the Structured Conversation Object using Pydantic models rather than
prompt text.

The Pydantic models become the canonical contract between the LLM provider and
the remainder of the platform.

## Reasoning

The platform separates semantic behavior from structural validation.

The prompt defines **what** should be extracted.

Pydantic defines **how** that information is represented.

Whenever the selected LLM provider supports native structured output, the
Pydantic-generated JSON Schema should be supplied directly to the provider
instead of embedding large JSON examples inside the prompt.

The same Pydantic models are then reused for:

- provider structured output
- runtime validation
- application types
- database persistence

This creates a single source of truth for every AI-generated artifact.

## Alternatives Considered

- Hand-written JSON examples inside prompts.
- Prompt-only schema enforcement.
- Independent validation models.

## Tradeoffs

Different LLM providers expose structured output differently.

The provider abstraction isolates those differences while allowing the
application to operate exclusively on validated `StructuredConversation`
objects.

The result is a provider-agnostic architecture with strongly typed AI
contracts.

# ADR-010 — Whole Conversation Processing

## Decision

Process each customer conversation as a single unit during Conversation
Understanding.

Do not chunk, summarize, or otherwise preprocess conversations before sending
them to the LLM.

Whole-conversation processing is the default strategy.

Chunking is considered a future scalability optimization rather than a
correctness requirement.

## Reasoning

The supplied dataset averages approximately thirteen messages per conversation,
with the longest conversations containing roughly fifty messages.

These conversations are comfortably within the context window of Gemini 2.5
Flash.

Providing the complete conversation preserves all available context while
avoiding cascading errors introduced by intermediate summarization or chunk
aggregation.

Conversation Understanding should perform exactly one semantic interpretation
of each conversation before producing the canonical Structured Conversation
Object.

## Alternatives Considered

### Conversation Chunking

Conversation

↓

Chunk

↓

Conversation Understanding

↓

Merge

Rejected because it introduces unnecessary complexity while increasing the
possibility of inconsistent issue extraction across chunk boundaries.

### Hierarchical Summarization

Conversation

↓

Summarization

↓

Conversation Understanding

Rejected because information is discarded before the canonical AI artifact is
created.

### Multi-pass Extraction

Conversation

↓

Issue Extraction

↓

Resolution Extraction

↓

Merge

Rejected because the current dataset does not justify the additional
complexity. A single-pass extraction is sufficient and considerably simpler.

## Tradeoffs

Very large conversations may eventually require chunking or hierarchical
processing.

If future datasets exceed practical context limits, chunking should be
introduced as a preprocessing stage without changing the canonical Structured
Conversation Object or any downstream consumers.

The remainder of the platform should remain unaware of whether Conversation
Understanding processed one conversation or multiple chunks.

# ADR-011 — Baseline-Derived Issue Catalog

## Decision

Generate the platform's Issue Catalog from the Day 1 baseline rather than
maintaining a manually curated taxonomy.

## Reasoning

The assignment explicitly defines Day 1 as the operational baseline.

Generating the taxonomy from the baseline aligns the platform with that
assumption while avoiding hardcoded issue definitions.

The catalog becomes operational data rather than application configuration.

Subsequent conversations classify against this baseline taxonomy.

Previously unseen issue categories become candidate anomalies.

The Issue Catalog is therefore treated as operational data rather than
application configuration.

It is generated by the platform rather than authored by developers.

## Alternatives Considered

- Hardcoded JSON taxonomy
- YAML configuration
- Hand-maintained issue catalog

## Tradeoffs

The baseline taxonomy is only as good as the Day 1 dataset.

A production system would likely introduce human review before promoting newly
discovered issue categories into the catalog, but that additional workflow is
outside the scope of this project.

---

## Taxonomy Usage

The Issue Catalog serves two independent architectural purposes.

### 1. Issue Normalization

During Conversation Understanding, extracted issues are normalized against the
existing Issue Catalog whenever an appropriate category already exists.

The prompt should preserve the customer's wording separately as
`customer_description`, while normalizing operational reporting through
`canonical_name`.

If no suitable category exists, the platform should create a new canonical
issue rather than forcing an incorrect match.

### 2. Novelty Detection

Issues that do not match the existing catalog become candidate novel issue
categories.

These unknown issue categories provide an additional anomaly signal alongside
traditional frequency-based anomaly detection.

This allows the platform to detect entirely new classes of operational issues,
not just increases in known ones.

### Ownership

The taxonomy belongs to the platform, not the LLM.

The LLM is responsible for extracting semantic information from conversations.

The platform is responsible for maintaining and evolving the operational
taxonomy over time.

This separation allows taxonomy evolution without coupling it to prompt
behavior or a specific LLM provider.

### Version 1 Canonicalization

For Version 1 of this project, canonicalization is performed directly by the
LLM during Conversation Understanding.

The prompt receives the current Issue Catalog and is responsible for
normalizing extracted issues into stable operational categories whenever an
appropriate category already exists.

This implementation intentionally favors simplicity over introducing a
dedicated taxonomy-matching service.

Future versions may move canonicalization into an independent platform
component without changing downstream consumers because the remainder of the
pipeline operates exclusively on the canonical Structured Conversation Object
and its relational projections.

# ADR-012 — Multi-Signal Anomaly Detection

## Decision

Treat anomalies as the result of multiple independent operational signals
rather than a single threshold.

Version 1 evaluates:

- volume spikes
- novel issue categories
- severity drift
- resolution drift

## Reasoning

Operational issues do not always manifest as simple increases in volume.

A previously unseen issue category or a dramatic change in support outcomes can
be equally important despite relatively low frequency.

Representing anomalies as collections of signals produces explanations rather
than opaque scores.

## Alternatives Considered

- Percentage threshold only
- Statistical outlier detection only
- LLM-generated anomaly detection

## Tradeoffs

This approach introduces additional rules but produces deterministic,
explainable anomalies while keeping the detection engine independent of the
LLM.

# ADR-013 — Conversation Isolation

## Decision

Treat one customer conversation as the fundamental unit of AI processing.

Each LLM invocation is responsible for producing exactly one
StructuredConversation from exactly one Conversation.

Do not batch multiple conversations into a single prompt.

If higher throughput becomes necessary, prefer provider-native batch execution
over changing the canonical processing model.

## Reasoning

The architecture is intentionally built around a one-to-one relationship:

Conversation

↓

StructuredConversation

↓

ConversationAnalysis

↓

ConversationIssue

Maintaining this invariant preserves:

- deterministic retries
- idempotent processing
- resumable pipeline execution
- precise audit trails
- simple progress reporting
- failure isolation

Every conversation remains independently executable, independently recoverable,
and independently observable.

## Alternatives Considered

### Multi-conversation prompts

Rejected because:

- a malformed response could invalidate multiple conversations
- retries become coarse-grained
- audit trails become ambiguous
- progress reporting becomes less precise
- conversation ordering becomes significant

### Provider-native Batch Execution

Preferred future optimization.

Provider-native batch APIs preserve the one-conversation processing model while
reducing network overhead and improving throughput.

## Tradeoffs

Processing one conversation per request results in more HTTP requests and
longer wall-clock execution.

However, the architecture remains significantly simpler and more robust.

Performance improvements should occur at the provider execution layer rather
than by changing the platform's fundamental processing model.

# ADR-014 — Deterministic Knowledge Synthesis

## Decision

Generate KnowledgeDocuments deterministically from the StructuredConversation
rather than invoking a second LLM.

The only AI operation in Phase 5 is embedding generation.

## Reasoning

Conversation Understanding has already performed the expensive semantic
reasoning required to understand the conversation.

Generating another LLM summary would duplicate work while introducing
additional latency, cost, and nondeterminism.

Instead, StructuredConversation becomes the canonical source for reusable
knowledge.

The application renders a deterministic knowledge_text representation suitable
for embeddings.

## Alternatives Considered

- Second LLM prompt for knowledge synthesis.
- Embedding raw conversations.
- Embedding ConversationAnalysis JSON directly.

## Tradeoffs

Template-generated knowledge is slightly less expressive than another LLM
pass but is perfectly reproducible, easier to test, cheaper to execute, and
keeps semantic reasoning isolated to Conversation Understanding.

# ADR-015 — Deterministic Evaluation

## Decision

Evaluate AI behavior using deterministic comparisons against curated
ground-truth datasets whenever structured outputs make this possible.

Avoid using an LLM as the primary evaluation mechanism.

## Reasoning

The platform already represents every AI stage using strongly typed canonical
artifacts:

- StructuredConversation
- ConversationIssue
- KnowledgeDocument
- ResolutionResponse

Because these artifacts are deterministic and schema-validated, they can be
compared directly against expected outputs without requiring another LLM.

This makes evaluation:

- deterministic
- reproducible
- inexpensive
- suitable for continuous integration
- appropriate for prompt regression testing

LLMs are the system under evaluation, not the evaluation framework.

## Alternatives Considered

### LLM-as-a-Judge

Rejected because:

- evaluation becomes nondeterministic
- results are difficult to regress across prompt versions
- evaluation incurs additional API cost
- explanations become harder to reproduce

LLM-based evaluation may be useful as a secondary qualitative signal, but not
as the platform's primary correctness metric.

## Tradeoffs

Deterministic evaluation cannot measure every qualitative aspect of AI
behavior.

However, it provides stable engineering metrics, supports automated
regression testing, and keeps evaluation independent of another language
model.

Human review remains appropriate for exploratory prompt development and
qualitative assessment, while deterministic evaluation serves as the canonical
engineering validation.

# ADR-016 — Embedding Document Rendering Strategy

## Decision

Render each `KnowledgeDocument` into a deterministic, human-readable
`knowledge_text` representation before generating embeddings.

The rendered document should emphasize the normalized operational issue and the
historical resolution while preserving customer wording as supporting semantic
context.

## Reasoning

Embeddings are intended to retrieve historical resolutions for semantically
similar customer problems.

The rendering therefore reflects the retrieval task rather than the storage
schema.

The goal is to maximize semantic similarity while remaining deterministic,
inspectable, and versionable.

### Canonical issue first

The normalized operational issue appears first because it represents the stable
concept the platform wants to retrieve.

Customer wording varies considerably across conversations, while canonical issue
names remain consistent over time.

### Preserve customer language

Customer symptoms are still included because they contain natural vocabulary
that future customer queries are likely to use.

This bridges operational taxonomy with real customer language.

### Resolution before symptoms

The historical resolution appears before the detailed symptom list because the
primary retrieval objective is to surface successful resolution paths.

Symptoms provide additional matching context after the core operational outcome
has been established.

### Human-readable rendering

The embedding text is rendered as plain text rather than JSON.

This improves readability during debugging, avoids embedding JSON syntax and
field names, and provides a stable representation that can evolve independently
from the stored schema.

### Exclude deterministic metadata

Metadata such as product, resolution type, and other structured fields are
available separately for deterministic filtering.

These fields do not need to dominate the embedding representation because they
are already handled by the retrieval pipeline.

## Alternatives Considered

- Embed raw JSON documents.
- Embed the Pydantic model directly.
- Generate embedding text using a second LLM prompt.

## Tradeoffs

A rendered representation requires maintaining a rendering template, but it is:

- deterministic
- inexpensive
- easily inspectable
- versionable
- optimized specifically for semantic retrieval

Changing the rendering strategy does not require changes to the canonical
`KnowledgeDocument` schema or downstream retrieval architecture.
