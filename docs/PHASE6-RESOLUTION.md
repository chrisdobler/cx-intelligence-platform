# Phase 6 — Resolution Assistant

## Goal

Provide grounded, explainable resolution recommendations using only historical
resolved knowledge.

The Resolution Assistant should never reinterpret raw conversations.

Instead it consumes the canonical artifacts generated by previous phases.

Pipeline:

Incoming Conversation
        │
        ▼
StructuredConversation
        │
        ▼
Metadata Filter
        │
        ▼
Vector Retrieval
        │
        ▼
ContextBundle
        │
        ▼
Resolution Assistant
        │
        ▼
ResolutionResponse

---

## Design Philosophy

The Resolution Assistant is a decision-support system.

It should recommend actions only when sufficient historical evidence exists.

It should never invent unsupported troubleshooting steps.

Grounded recommendations are preferred over speculative answers.

---

## ContextBundle

The Context Builder is deterministic.

It gathers:

- the current issue
- retrieved KnowledgeDocuments
- retrieval metadata

It performs no AI reasoning.

The ContextBundle becomes the complete input to the Resolution Assistant.

---

## ResolutionResponse

The Resolution Assistant returns a structured response.

Suggested fields:

- recommendation
- reasoning
- recommended_actions
- grounded
- evidence_strength
- citations

Every recommendation should include citations to the KnowledgeDocuments that
support it.

---

## Grounding Policy

Recommendations must be based exclusively on retrieved KnowledgeDocuments.

If retrieval does not provide sufficient evidence, the assistant should return
an ungrounded response rather than inventing a recommendation.

Example:

"No sufficiently similar historical resolutions were found."

This is considered a successful outcome.

---

## Retrieval Strategy

Retrieval occurs before the Resolution Assistant.

The assistant never performs retrieval itself.

The assistant reasons only over the ContextBundle supplied by the platform.
